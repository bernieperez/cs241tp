Chapter 1.     Introduction

This tutorial contains examples and explains how a JVI program using the transparent package can

  • access and modify objects created from another language
  • create new persistent objects for a schema defined from another language.

We assume that you have some general familiarity with writing Transparent Java programs using JVI. At the least, you need to be comfortable with the extent of JVI usage covered in Chapter 3 Tutorial for Transparent Package in the JVI Usage Manual.

In presenting C++ examples, we assume that you are familiar with using the VERSANT C++ language interfaces to create and access persistent objects using C++.

Chapter 2.     Basic Concepts

2.1.   Limitations

There are two basic assumptions that we work with when sharing objects with a Transparent JVI program.

  • The object sharing is one-directional. This means your Transparent JVI program can access database schema for a class created from a language other then Java. We do not address the converse - sharing the schema of a class defined from a Transparent JVI program with other languages is not considered.
  • We provide support for transparent sharing of the attributes (data fields) of objects, not their methods. However, you may write methods in Java for the external classes being shared. The is described in the course of this tutorial.

2.2.   The Java Foreign View Approach

Foreign Language

The term Foreign refers to the external language from which the schema of a class is defined. Currently, the foreign language we support for object sharing with JVI is

  • C++

Transparency and Object Sharing

Recall that to create a persistent Java class using JVI you first create a .java file that defines and implements the persistent class. Then you compile and enhance the class file. The enhancement process automatically incorporates changes needed to allow objects of the class to be stored in the database.Before running the Enhancer please make sure the CLASSPATH is set properly.

In the Object Sharing support we provide, we work under the same paradigm. Our goal is to provide the same level of convenience through transparent access to schema defined from a foreign language as we do for class schema defined from Java.

An object sharing support tool creates Java classes, by providing you with .java files. These contain source code for the classes defined to the database from a foreign language. You can then use these classes and compile and enhance them together with your Transparent JVI program. You can treat the class created for the foreign language schema just like any another persistent Java class.

To start a session in transparent JVI applications for java foreign view approach instead of a default standard session, set the session property by name "useJFV" to value "true" when you create a session object.

View and Replacement Classes

A JVI programmer can use the Db2Java tool to generate Java files from the database corresponding to class schema defined from a foreign language. This tool, com.versant.Db2Java is a Java program that reads the database schema and creates two .java source files. One file corresponds to the transparent class being shared. We call this the View class because it presents a Java JVI programmer a Java-centric view of the class being shared.

The other file that we generate is called the Replacement class. This class bears a close correspondence with the View class. It encapsulates a direct correspondence with the schema present in the database. While the View class contains attribute names and types that a Java and JVI programmer would be familiar with, the Replacement class contains attributes aligned with their type in the database. The Replacement class also contains methods to transform the attributes of the View class with the attributes of the Replacement class and vice-versa.

For most cases of object sharing, you need to be familiar with the structure of the View class. The Replacement class serves to read and write shared objects from and to the database correctly. You must refer to the View class in your transparent JVI program. When you compile your application, you must also compile the View and Replacement classes. The enhancement must also include the compiled View and Replacement classes together with your application classes.

Chapter 4.     C++: Sharing The Vehicle Class with JVI

4.1.   Generating the View and Replacement Class

We have created the schema for the Vehicle class and created a persistent instance of the class from the C++ language interface. The tutorial now illustrates how a Transparent JVI program can access instances of this class and create new ones.

The first stage in developing the Transparent JVI application is to generate View and Replacement classes for the schema defined from C++.

4.2.   Using com.versant.Db2Java

The Java program com.versant.Db2Java generates Java source for C++ defined schema stored in the VERSANT database. This command accepts several options, and these are itemized in the com.versant.Db2Java Usage Notes chapter presented later on in this tutorial. 

  -d <dbname>

The -d option accepts the database from which the schema is to be read. In this example, the database would be cppdb. You must always specify a database name so that the tool is aware of which database to read schema from. 

  -verbose

The -verbose option prints out pertinent information while generating the Java View and Replacement class source code. If you have already generated source code, you need to use the -force option to clobber the already generated files since by default generated code will not be overwritten. 

  -class <classname1> <classname2> ....

You may also specify which files to generate from the database through using the -class option. Only those foreign language classes that match the specified names against the -class option will be generated. The matching conforms to Perl5 regular expression matching syntax for the specified patterns. If you omit this option, all foreign language classes from the database would be generated, i.e. its equivalent to specifying a wild card pattern match.

By default, when classes are generated according to the names specified against the -class option, their super classes if any are also generated. You can explicitly curb this behavior through specifying the -nosuper option. We do not expect that you would want to do this unless you have changed a sub-class and only want to regenerate Java code for this class in a second pass of running the program. 

  -o <output path>

The -o option specifies in which directory you want the Java files generated. As mentioned, you need to integrate the generated Java View and Replacement classes with your Transparent JVI program that uses them. So, you would be eventually compiling and enhancing these classes. A good place to be generating these Java classes would be in your enhancer input directory. Following compilation with the rest of your application, this will allow you to enhance these classes with the rest of your application classes.

In the absence of specifying the -o <output path> option, Java source code is generated in the same directory from which the com.versant.Db2Java program is run. The source code is generated in a directory named "jfv" (an abbreviation for "Java Foreign View"). Each of the generated classes bears the jfv. package name. This allows for clean namespace separation of foreign view classes.

4.6.   Copying Prior to Pickling and After Unpickling

The Replacement class transforms attributes when they are being read from and written to the database so that the View class always presents the specified attribute names and types to your application. The PObject and PVirtual Replacement Classes contain methods that do not perform any transformation of attributes because we are not concerned with any attributes in these classes. The Vehicle_Rep Java source code illustrates these transformations. JVI uses the term Pickling to refer to writing a Java object into the database. Unpickling refers to the converse, where a persistent Java object has its values read in from the database.

Every Replacement class will contain the copyPrePickle and copyPostUnpickle methods.

copyPrePickle copies an object, attribute by attribute from the View object into a Replacement object before the Replacement object is stored (pickled) to the database.

copyPostUnpickle copies an object, attribute by attribute from the Replacement object into the View object when the object is read (unpickled) from the database.

In the Vehicle example we consider, the Replacement class has the same attribute types as the View class for the year and retailCost. The "transformations" for these attributes are therefore trivial. copyPrePickle copies attributes directly from the View to the Replacement instance. copyPostUnpickle copies attributes directly from the Replacement to the View instance. However, the manufacturer and model are presented as Strings in the view class, but are byte arrays in the Replacement class. Hence, the utility functions:

JfvUtilities.CPP_getByteArrayFromString (java.lang.String aString)

and

JfvUtilities.CPP_getStringFromByteArray (byte[] byteArray);

are used to transform the attribute from the View to the Replacement class during pickling and from the Replacement to the View class during unpickling, respectively.

For storage transforms and object sharing involving attribute types in the database that do not have a direct correspondence to View class attributes, the transformations are not trivial. This is especially true in the case of using embedded C++ collections in the C++ classes. We will illustrate these transformations through examples later on in this tutorial.

4.7.   The Transparent JVI Application

You are ready to use the Vehicle class in a Transparent JVI application program. The application has already been provided to you in the <VERSANT_ROOT>/demo/jvi/advanced/sharing/cpp/in directory. The file Example1.java illustrates how a Transparent JVI program could use the Vehicle class.

In the example program, we select all Vehicle instances from the database and increment the Vehicle's retail cost by $1000.00. This select is done through executing a VQLQuery. Then, we create a new Vehicle instance and store it in the database.
  import com.versant.trans.*;
  import com.versant.fund.*;
  import java.util.*;

  public class Example1 {

    public static void main (String[] args)
    {
      if (args.length < 1) {
        System.err.println ("Usage: java Example1 <dbname>");
        System.exit (1);
      }

      Properties p = new Properties ();
      p.setProperty ("database", args[0]);
      p.setProperty("useJFV" , "true");
      TransSession s = new TransSession (p);

      VQLQuery query = new VQLQuery (s, "select selfoid from Vehicle");
      Enumeration e = query.execute ();

      if (!e.hasMoreElements()) {
        System.out.println ("No Vehicle instances were found.");
      }
      else {
        while ( e.hasMoreElements() ) {
          jfv.Vehicle v = (jfv.Vehicle) e.nextElement ();
          System.out.println ("Found Vehicle instance with\n" +
                              "\tManufacturer: " + v.manufacturer + "\n" +
                              "\tModel: " + v.model + "\n" +
                              "\tYear: " + v.year + "\n" +
                              "\tRetailCost: " + v.retailCost);

          // increment the Vehicle's retailCost by 1000.00
          v.retailCost += 1000.00;
        }
      }

      // create a new Vehicle instance and make it persistent
      jfv.Vehicle v = new jfv.Vehicle ();
      v.manufacturer = "Audi";
      v.model = "A300";
      v.year = 1998;
      v.retailCost = 32650.0;

      s.makePersistent (v);
      s.commit ();
    }
  }

6.5.   Copying Prior to Pickling and After Unpickling

The Car_Rep Java source code contains the copyPrePickle and copyPostUnpickle methods that illustrate transformations between the View class attributes and the Replacement class attributes. Recall that copyPrePickle copies an object, attribute by attribute from the View instance into a Replacement instance before the object is stored (pickled) to the database. copyPostUnpickle copies an object, attribute by attribute from the Replacement instance into a View instance when the object is read (unpickled) from the database.

In the Car class, the doors and cylinders attributes have the same type in the Replacement and View class. As in the case of Vehicle the "transformations" are therefore trivial. However, the o_u4b attribute which is a long in the view class goes through a transformation. The id attribute in Car is stored in the database as an unsigned 4 byte value. int in Java is 4 bytes, which means we need a larger sized type to store the id. The nearest larger type is long. The database type o_u4b hence maps to the type long in the transparent Java language binding of the View class. Since the range of Java long is larger than that of o_u4b which is the type in the database, there could be a loss of information when storing a value from Java. You need to be aware of this when you write your object sharing programs with JVI.

The transformations for the id attribute are therefore as follows:

    • When copying from the View class (where the attribute is a long) to the Replacement class (where the attribute is an unsigned integer, int) we call the utility method JfvUtilities.CPP_getIntFromLong.
    • When copying from the Replacement class (where the attribute is an unsigned integer int) to the View class (where the attribute is a long) we call the utility method JfvUtilities.CPP_getLongFromInt.

JfvUtilities is a class in the com.versant.utils package which contains a large number of transformation functions named according to the transformation being performed.

10.5.   Copying Prior to Pickling and After Unpickling

The FloorPlan_Rep Java source code contains the copyPrePickle and copyPostUnpickle methods that illustrate transformations between the View class attributes and the Replacement class attributes. Recall that copyPrePickle copies an object, attribute by attribute from the View instance into a Replacement instance before the object is stored (pickled) to the database. copyPostUnpickle copies an object, attribute by attribute from the Replacement instance into a View instance when the object is read (unpickled) from the database.

In the FloorPlan class, the name attribute is a byte array in the replacement class and a String in the view class, hence the utility methods in JfvUtilities.CPP_getByteArrayFromString (String) and JfvUtilities.CPP_getStringFromByteArray (byte[]) are called during pickling and unpickling the view class from the database. The cubesAssigned attribute which is of type com.versant.util.CppHashMap in the view class goes through a more complex transformation. The four attributes in the replacement class (and the database) which represent the VEIDictionary contents are mapped to the CppHashMap attribute in the View class, and appropriate JfvUtility methods used in transformations. The following methods return keys o_4b[] values PObject[] table o_4b[] and links o_4b[] from the CppHashMap argument during pickling: JfvUtilities.CPP_get4bKeysArrayFromCppHashMap (CppHashMap) JfvUtilities.CPP_getValuesArrayFromCppHashMap (CppHashMap) JfvUtilities.CPP_getTableArrayFromCppHashMap (CppHashMap) JfvUtilities.CPP_getLinksArrayFromCppHashMap (CppHashMap) For unpickling, the JfvUtility method: JfvUtilities.CPP_getCppHashMapFrom4bArray (o_4b[], PObject[], o_4b[], o_4b[]) returns a CppHashMap from the database attributes representing a VEIDictionary.

10.6.   The Transparent JVI Application

You are ready to use the FloorPlan class in a Transparent JVI application program. The application has already been provided to you in the <VERSANT_ROOT>/demo/jvi/advanced/sharing/cpp/in directory. The file Example3.java illustrates how a Transparent JVI program could use the FloorPlan class.

In the example program, we select all FloorPlan instances from the database. Then we print out all the cube assignments (i.e., the cube number with the employee assigned to the cube). This is done by obtaining an iterator on the keys of the CppHashMap through calling the CppHashMap.keys () method. The iterator is traversed using the CppHashMapIterator.hasNext () and CppHashMapIterator.next () methods. The value mapped against a key in the CppHashMap (i.e. the Employee instance in this example) is obtained based on the key using the CppHashMap.get (Object key) method.

We also create a new assignment of a cube to an employee before commiting the session.
import jfv.*;
import com.versant.trans.*;
import com.versant.fund.*;
import com.versant.util.*;
import java.util.*;

public class Example3
{
  public static void main (String[] args) {
    if (args.length < 1) {
      System.err.println ("Usage: java Example3 <dbname>");
      System.exit (1);
    }

    Properties p = new Properties ();
    p.setProperty ("database", args[0]);
    p.setProperty("useJFV" , "true");
    TransSession s = new TransSession (p);

    VQLQuery query = new VQLQuery (s, "select selfoid from FloorPlan");
    Enumeration e = query.execute ();

    if (!e.hasMoreElements()) {
      System.out.println ("No FloorPlan instances were found.");
    }
    else {
      while ( e.hasMoreElements() ) {
        jfv.FloorPlan fp = (jfv.FloorPlan) e.nextElement ();
        CppHashMapIterator keys = fp.cubesAssigned.keys ();
        while (keys.hasNext ()) {
          Object key = keys.next ();
          jfv.Employee emp = (jfv.Employee) fp.cubesAssigned.get (key);
          System.out.println ("Cube: " + key +
                              "  Employee: " +
                              "Name: " + emp.name + "  Id: " + emp.id);
        }
        jfv.Employee newEmp = new Employee ();
        newEmp.name = new String ("Gary Stevenson");
        newEmp.id = 108;
        s.makePersistent (newEmp);
        fp.cubesAssigned.put (new Integer (20), newEmp);
      }
    }

    s.commit ();
  }
}
